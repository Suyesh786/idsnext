Abstract Data Type ADT — What & Why

01

What is an Abstract Data Type (ADT)?

🧩
An Abstract Data Type (ADT) is a mathematical model for a data type, defined by its data values and the operations that can be performed on it — completely independent of any implementation details.

Think of an ADT as a contract or specification. It tells you what a data structure does, not how it does it. The internal mechanics — arrays, pointers, memory layout — are hidden from the user. This separation is the essence of abstraction.

For example, a Stack ADT defines that you can push, pop, and peek. Whether it is built using an array or a linked list internally is irrelevant to the programmer using it.

02

Real World Analogies

🏧

ATM Machine

You interact with an ATM using buttons: withdraw, deposit, check balance. You don't know how the bank's internal system works — you only use the defined interface.

📺

TV Remote

A TV remote gives you operations: volume up, channel change, power. You never need to understand the IR signals or circuit boards inside — just use the buttons.

🚗

Car Driving Interface

A driver uses the steering wheel, pedals, and gear stick. Whether the car is petrol, diesel, or electric is hidden. The driving interface (ADT) stays the same.

03

Characteristics of ADT

Encapsulation

Data and the operations on it are bundled together. The internal state is hidden and can only be accessed through defined operations.

Principle Data Hiding
Abstraction

The user works with a simplified view — only the what, never the how. Implementation details are completely invisible.

Principle Interface Only
Defined Operations

Every ADT comes with a clear, well-defined set of operations. These form the contract between the user and the implementation.

Principle Fixed API
Implementation-Free

The same ADT can have multiple implementations. A List ADT can be built with an array or a linked list — both satisfy the contract.

Principle Interchangeable

04

Types of ADTs

ADTs are broadly classified into two categories based on how their elements are organised and the relationships between them.

Linear ADTs Sequential structure

Elements are arranged in a single sequence, one after another. Each element has at most one predecessor and one successor.

Array ADT Fixed-size, indexed, random access in O(1)
List ADT Dynamic sequence — insert and delete anywhere
Stack ADT LIFO — last inserted is first removed
Queue ADT FIFO — first inserted is first removed
Non-Linear ADTs Hierarchical / networked

Elements can have multiple predecessors or successors, forming hierarchical or networked relationships.

Tree ADT Hierarchical structure — one root, many children
Graph ADT Nodes connected by edges — general network model
Heap ADT Tree satisfying heap property (min or max)
Map / Set ADT Key-value associations or unique-element collections
05

Example Implementation in C

Below is a simple List ADT in C — demonstrating how the ADT specification (the struct and its operation signatures) separates from the implementation details inside each function body.

The specification defines the data (the struct) and the operation signatures. This is what users of the ADT see.

C — List ADT Specification (header) 
/* list_adt.h — Public interface. Users only see this. */

#define MAX_SIZE 100

/* ── Data Definition ────────────────────────────── */
typedef struct {
    int  data[MAX_SIZE]; /* storage — hidden from user */
    int  size;           /* current number of elements */
} ListADT;

/* ── Operation Signatures (the ADT contract) ─── */
void list_init   (ListADT* L);
void list_insert (ListADT* L, int pos, int value);
void list_delete (ListADT* L, int pos);
int  list_get    (ListADT* L, int pos);
int  list_size   (ListADT* L);
void list_display(ListADT* L);
ℹ️ This header file is the ADT. A programmer can use the List without ever looking at the .c implementation file.

The implementation provides the actual code for each operation. It could be swapped for a linked-list version without the user noticing.

C — List ADT Implementation 
/* list_adt.c — Internal implementation. Users don't see this. */

void list_init(ListADT* L) {
    L->size = 0;
}

void list_insert(ListADT* L, int pos, int value) {
    if (L->size >= MAX_SIZE || pos < 0 || pos > L->size) return;
    for (int i = L->size; i > pos; i--)
        L->data[i] = L->data[i - 1];  /* shift right */
    L->data[pos] = value;
    L->size++;
}

void list_delete(ListADT* L, int pos) {
    if (pos < 0 || pos >= L->size) return;
    for (int i = pos; i < L->size - 1; i++)
        L->data[i] = L->data[i + 1];  /* shift left */
    L->size--;
}

int list_get(ListADT* L, int pos) {
    return (pos >= 0 && pos < L->size) ? L->data[pos] : -1;
}

int list_size(ListADT* L) { return L->size; }

void list_display(ListADT* L) {
    printf("List: [ ");
    for (int i = 0; i < L->size; i++)
        printf("%d ", L->data[i]);
    printf("]\n");
}
ℹ️ This implementation uses an array internally. You could replace it with a linked list and the user's code would work unchanged — that is the power of ADTs.
06

Advantages & Limitations

🔒

Information Hiding

Users only interact through the defined interface. Internal details are shielded, reducing unintended misuse and coupling.

♻️

Reusability

An ADT can be implemented once and reused across many programs. The same Stack ADT works in expression evaluators, compilers, and browsers.

🔄

Implementation Flexibility

You can swap the underlying implementation (array ↔ linked list) without breaking any existing user code — only the .c file changes.

⚙️

Overhead

Function call indirection and abstraction layers can introduce slight performance overhead compared to direct, implementation-aware manipulation.

📐

Design Complexity

Designing a clean, complete ADT interface requires careful thought upfront. A poorly designed ADT is hard to extend later without breaking contracts.

🧪

Not Directly Runnable

An ADT is a specification — it cannot be run or tested on its own. It always requires at least one concrete implementation before it becomes usable.

07

Applications of ADTs

🏗️

Software Design

ADTs are the backbone of modular software. Each module exposes an ADT-style interface — enabling teams to build, test, and swap components independently.

🗂️

Data Organisation

Databases and file systems use ADTs (like Maps, Sets, Queues) to organise records and buffers, abstracting the storage engine from the query layer.

📦

Standard Libraries

Python's list, C++'s std::vector, and Java's ArrayList are all concrete implementations of the List ADT — ready to use out of the box.

🌐

Operating Systems

OS kernels use Queue ADTs for process scheduling, Stack ADTs for function call management, and Graph ADTs for resource allocation graphs.
